## 3. Product Requirements (PRD View)
Project: MultiSow v2, AI-enhanced multi-tier (stratified) crop planning platform.

Design philosophy: **Indian Farmer First** — every UI surface, unit, currency symbol, and icon is optimized for Indian smallholder farmers.

Core user problem:
- Farmers/planners need high-yield, resource-efficient intercropping design guidance, not single-crop planning.

Primary product goals:
- Design and simulate 4-layer intercropping systems
- Provide preset region-specific model templates (6 Indian regional presets)
- Offer AI advisory (rule-based + LLM NLP) with farmer-friendly language
- Offer ML-assisted prediction, optimization, and explainability (FOHEM)
- Support practical farm operations (spacing, compatibility, economics)
- Display all measurements in Indian units: acres, cents, ₹ (INR with lakhs/crores formatting)
- Use universally recognizable Google Material Symbols instead of abstract icon libraries

Primary users:
- Indian smallholder and medium farmers
- Farm consultants/agronomists
- Experimental/research users testing multi-tier models

Expected value:
- Better Land Equivalent Ratio (LER)
- Better yield per area (tonnes per acre)
- Improved water/resource efficiency
- Better crop compatibility and lower conflict risk
- Revenue projections in ₹ lakhs with realistic Indian market pricing

### 6.4 Icon System Architecture
All icons across the entire frontend use **Google Material Symbols** (variable font), loaded via CDN stylesheet in `app/layout.tsx`. Zero `lucide-react` imports remain in any source file.

**Icon component** (`components/ui/Icon.tsx`):
- Props: `name: string` (Material Symbol name), `size?: number` (default 24), `filled?: boolean`, `weight?: 100–700`, `className?: string`
- Renders `<span className="material-symbols-outlined">` with CSS `font-variation-settings` for FILL, wght, GRAD, opsz
- All component icon props throughout the codebase accept `string` (Material Symbol name), not component references

**Icon prop API pattern** — shared components use `icon?: string`:
- `Button.tsx`: `leftIcon?: string`, `rightIcon?: string`
- `Badge.tsx`: `icon?: string`
- `Input.tsx`: `leftIcon?: string`, `rightIcon?: string`
- `Panel.tsx`: `PanelHeader.icon?: string`, `Section.icon?: string`, `SectionHeader.icon?: string`
- `StatCard` (MetricsDashboard): `icon: string`
- All callers pass Material Symbol name strings: `icon="agriculture"`, `leftIcon="search"`, etc.

### 6.5 Indian Unit System
`lib/units.ts` provides all measurement conversion and formatting functions used across the platform:

**Constants:**
- `CENTS_PER_ACRE = 100`
- `M2_PER_CENT = 40.4686`
- `M2_PER_ACRE = 4046.86`

**Conversion functions:**
- `acresToCents(acres)` — acres → cents (rounded)
- `centsToAcres(cents)` — cents → acres
- `acresToM2(acres)` — acres → square metres
- `centsToM2(cents)` — cents → square metres
- `acresToFarmDimensions(acres)` → `{ widthM, lengthM, areaM2, cents }` (square farm)
- `metersToCents1D(meters)` — linear metres → 1D cent equivalent
- `thaToTonnesPerAcre(t_ha)` — tonnes/hectare → tonnes/acre
- `thaToTotalTonnes(t_ha, acres)` — total tonnes for a given farm area

**Formatting functions:**
- `formatArea(acres)` → `"2.5 acres (250 cents)"`
- `formatYield(t_ha, acres)` → farmer-friendly yield text with total tonnes
- `formatMoney(rupees)` → `"₹12.5 lakhs"` or `"₹1.2 crores"` (Indian numeral grouping)
- `formatSpacing(meters)` → `"8.0 meters apart"`
- `formatLER(ler)` → `"68% more productive than monoculture"`
- `getGridLabelCents(farmAcres)` → `{ metersPerLine, label }` for cent-scale grid labels

All dashboard metric cards, designer panels, prediction results, and revenue projections use these functions.

### 6.6 Design Token System
Token-first approach using CSS custom properties:

1. **`styles/tokens.css`** — raw design tokens as CSS variables:
   - Primary color scale (50–950, green tones)
   - Dark theme backgrounds: `--color-bg-base: #0A0F0A` through `--color-bg-muted`
   - Text colors, border colors (with rgba), accent colors (earth, sky)

2. **`app/globals.css`** — imports `tokens.css`, then declares Tailwind `@theme` block:
   - Bridges CSS tokens into Tailwind: `--color-background`, `--color-surface`, `--color-earth`, `--color-sky`, font families
   - Base styles: body uses token colors/fonts
   - Typography heading rules (h1–h6) using DM Sans display font with tight tracking

3. **Consumed everywhere** via Tailwind utility classes that map to token values.

## 21. Domain Glossary
- Strata / Layering: Designing crops in vertical layers so light, root depth, and nutrient zones are better utilized.
- LER (Land Equivalent Ratio): Ratio comparing intercropping productivity to monocropping; `> 1` means intercropping advantage.
- LAI (Leaf Area Index): Leaf area per ground area; influences light interception.
- k coefficient: Beer-Lambert extinction coefficient; higher means stronger light attenuation.
- VWC: Volumetric water content in soil.
- GDD: Growing degree days, thermal accumulation for growth progression.
- ROI (Root Overlap Index): Root zone overlap metric, used for belowground competition.
- FOHEM: Fuzzy-Optimized Hybrid Ensemble Model (FIS + RF + CatBoost + ELM + GA weighting/selection).        
- Pareto front: Non-dominated set of multi-objective optimization solutions.

## 22. End-To-End Workflow Narratives
### 22.1 Farmer design workflow
1. User opens `/designer`.
2. Chooses species/layers and places plants or loads presets. Farm area is entered in acres with automatic cents conversion.
3. Frontend computes spacing/compatibility overlays via `lib/*` and `hooks/*`.
4. The 3D canvas shows a cent-labeled grid (`CentGrid` with √40.4686m spacing) and the FarmHUD displays acres/cents readout.
5. User clicks **Apply & Predict** button to trigger a manual prediction run via `useFohem` hook.
6. User can move to `/predict` for detailed yield/interpretability and `/optimize` for recommendations.

### 22.2 Prediction workflow (intended)
1. `components/ml/YieldPredictionTool.tsx` sends request to `/ml/predict`.
2. FastAPI ML router should transform layer inputs into FOHEM-ready features.
3. FOHEM layer models return per-layer yield, confidence intervals, and feature explanations.
4. Response powers `ExplainPredictionTool` and dashboard visuals.

Current status caveat:
- Step 2 and 3 work via heuristic physics-based fallback in the current `ml_routes.py /ml/predict` endpoint. When FOHEM is trained (using the `/ml/train` endpoint), the system switches to model-based prediction automatically.

### 22.3 AI planning workflow
1. Frontend posts planning payload to Next proxy `app/api/ai/plan/route.ts`.
2. Proxy forwards to backend advisor endpoint.
3. `AIStratificationAdvisor.generate_full_plan` scores preset candidates using budget/soil/goal heuristics.
4. Backend returns selected model, budget split, expected revenue range, tier plan, and apply payload.

## 26. Algorithmic Detail Summary
### 26.1 FOHEM training logic
1. Compute FIS stress outputs from agronomic inputs.
2. GA feature selection chooses informative subset.
3. Train RF, CatBoost, ELM on selected features plus stress augmentation.
4. Use GA to optimize ensemble weights minimizing validation MAE.
5. Predict with weighted blend and optional confidence intervals.

### 26.2 FIS logic
- Inputs include LAI, soil N, VWC, root overlap, shade fraction, pH, spacing, GDD, rainfall, solar angle, density.
- Output stress dimensions include canopy light competition, nutrient competition, water stress.
- Supports skfuzzy mode and parametric fallback mode.

### 26.3 Strata optimizer logic
- Multi-objective search over crop selections and spacing.
- Objectives: maximize yield and LER, maximize profit, minimize competition.
- Uses non-dominated sorting and crowding distance selection.

## 39. ML Inference Module (`multisow/ml/inference/`)

### 39.1 Overview
Previously, crop recommendation inference was handled inline inside the router. It is now separated into a dedicated inference module for cleaner dependency management and testability.

### 39.2 `crop_recommender.py`
Three public async functions:
- `recommend_crop(N, P, K, temperature, humidity, ph, rainfall)` — loads GaussianNB artefacts lazily, runs StandardScaler transform, returns predicted crop + top-N compatible intercropping partners + confidence.
- `recommend_for_layer(layer, ...)` — filters the recommendation through `get_crops_by_layer` to return only layer-appropriate crops.
- `extend_with_ai(crop_list, farm_context)` — calls Anthropic API to attach a natural-language description and planting advice to each recommended crop; falls back to static descriptions when API unavailable.

Artefacts are cached at module level after first load (`_model`, `_scaler`, `_labels` module globals).

### 39.3 `multisow/ml/config.py`
`MLSettings` is a Pydantic BaseSettings class that loads all ML subsystem configuration from environment variables or `.env`:
- `DATABASE_URL`, `INFLUXDB_*`, `POSTGRES_URL`, `REDIS_URL`, `MLFLOW_*`
- `ANTHROPIC_API_KEY`, NLP model preferences
- Feature engineering toggles

Single import `from multisow.ml.config import settings` replaces the scattered `os.getenv()` calls that existed before.

### 33.1 Primary crop dataset
**Location:** `multisow/ml/data/crop_recommendation/`

Two CSV files are the single source of truth for all crop intelligence features:

| File | Rows | Columns |
|---|---|---|
| `crop_data1.csv` | 115 | N, P, K, temperature, humidity, ph, rainfall, label |
| `crop_data2.csv` | 115 | N, P, K, temperature, humidity, ph, rainfall, label |

Both files share an identical schema. After deduplication (on feature columns), the combined library contains approximately 115 unique agronomic profiles across 22 crop classes.

**Feature definitions:**
- `N` — Nitrogen ratio in soil (kg/ha)
- `P` — Phosphorus ratio in soil (kg/ha)
- `K` — Potassium ratio in soil (kg/ha)
- `temperature` — Mean ambient temperature (°C)
- `humidity` — Relative humidity (%)
- `ph` — Soil pH value
- `rainfall` — Annual / seasonal rainfall (mm)
- `label` — Crop class name (target variable)

**22 supported crop classes:**
apple, banana, blackgram, chickpea, coconut, coffee, cotton, grapes, jute, kidneybeans, lentil, maize, mango, mothbeans, mungbean, muskmelon, orange, papaya, pigeonpeas, pomegranate, rice, watermelon.

**Critical note:** No pre-trained model artefacts are committed to the repository. Before ML prediction endpoints are live, you must run:
```
py -m multisow.ml.pipelines.train_crop_recommender
```
This saves three `.joblib` artefacts to `multisow/ml/models/saved/`:
- `crop_recommender_model.joblib` (GaussianNB — best performer)
- `crop_recommender_scaler.joblib` (StandardScaler fitted on training split)
- `crop_labels.joblib` (sorted unique label list)

### 33.2 Crop library module (`multisow/ml/data/crop_library.py`)
Loaded **once at import time**, this module deduplicates both CSVs and exposes a pure Python helper API used by three consumers: the inference engine, the preset generator, and the FastAPI crop-library router.

Key public functions:
- `get_all_crops()` — sorted list of all 22 crop label strings
- `get_crop_stats(crop_name)` — mean feature dictionary for one crop
- `get_all_crop_stats()` — all crop stats in one call
- `find_compatible_crops(crop_name, top_n=5)` — returns top-N most compatible crops using normalised L1 feature distance
- `get_crops_by_layer(layer)` — maps crops to intercropping layers ("canopy", "midstory", "understory", "groundcover") based on temperature and nitrogen profiles

**Compatibility scoring:**
Distance between crops `a` and `b` is:
$$\text{distance}(a, b) = \sum_{f} \frac{|\bar{f}_a - \bar{f}_b|}{\sigma_f}$$
Lower score = more similar growing requirements = more compatible for intercropping.

**Layer assignment rules:**
| Layer | Temperature condition | N condition |
|---|---|---|
| canopy | mean > 27 °C | mean N > 60 |
| midstory | 20–27 °C | N 30–60 |
| understory | < 20 °C | or N < 30 |
| groundcover | any | N < 20 AND humidity > 70% |

### 33.3 Crop Recommender training pipeline (`multisow/ml/pipelines/train_crop_recommender.py`)
Trains two classifiers on an 80/20 train/test split and saves the better one:
1. **GaussianNB** (primary) — probabilistic, fast inference, typically wins on this dataset
2. **SVC** (secondary) — support vector classifier with RBF kernel

Both are feature-scaled with `StandardScaler` first. GaussianNB is saved as the default model since it performs better.

Accuracy on the 80/20 split is typically >95% on both classes given the clean, compact dataset.

### 33.4 Preset generator (`backend/data/preset_generator.py`)
Programmatically builds the 6 intercropping preset models shown to users in the frontend (strata browser, AI advisor, designer):

1. Selects one representative crop per layer (canopy, midstory, understory, groundcover) from the dataset.
2. Ranks all 4-crop combinations by sum of pairwise compatibility distances (lower = better harmony).
3. Takes top-6 combos and materialises them into structured preset dicts with field-ready metadata (soil hints, region hints, spacing, budget, yield estimates).

The presets power `backend/ai_advisor.py`'s `generate_full_plan()` and are served at `GET /presets`.

### 33.5 Ingestion pipeline (`multisow/ml/pipelines/ingest.py`)
Handles four real-world data sources:

| Source class | Input | Transport |
|---|---|---|
| `IoTSensorIngester` | InfluxDB 2.x time-series (soil moisture, canopy temp, leaf wetness) | InfluxDB client; falls back to in-memory |
| `ManualRecordIngester` | CSV uploads (planting, fertiliser, harvest events) | Pandas read_csv |
| `ClimateFeaturesIngester` | NASA POWER daily weather API | HTTP; falls back to IMD-style mock |
| `DroneSatIngester` | GeoTIFF NDVI rasters | rasterio (optional); falls back to random stub |

All four return standardised Pandas DataFrames for downstream preprocessing.

### 33.6 Preprocessing pipeline (`multisow/ml/pipelines/preprocess.py`)
`StratifiedIntercroppingPreprocessor` takes fused DataFrames from ingestion and applies:
- **Normalisation:** MinMax (default) or Z-score per column, configurable per feature
- **Imputation:** KNN for sparse static fields; cubic spline for IoT time-series gaps
- **Feature engineering:** Beer-Lambert canopy opacity, GDD accumulation, PAR statistics, root overlap index (ROI), ETo estimation
- **Multi-source fusion:** Aligns sensor, climate, manual, and drone frames into a single daily feature matrix with a MD5 fingerprint hash for deduplication

### 33.7 Feature store (`multisow/ml/pipelines/feature_store.py`)
`FeatureStore` persists versioned feature matrices for reproducible model training:
- Dev mode: Parquet files on disk under `multisow/ml/feature_store_data/`, metadata in SQLite
- Prod mode: Parquet on shared volume/object store, metadata in PostgreSQL
- Each version is tagged with a UUID, timestamp, feature list, and row count
- Supports retrieval by version ID or latest tag

### 33.8 Frontend planting guide data (`data/planting-guides.ts`)
Static TypeScript dataset providing rich operational guidance for each of the 6 regional preset models. Data structures per guide:
- `PlantingSequenceStep[]` — week-by-week activity schedule per layer/species
- `SpacingGuide[]` — row/plant spacing, depth, and planting method per species
- `IrrigationSchedule[]` — stage-based frequency, method, and volume
- `FertilizerSchedule[]` — monthly fertiliser application with quantity and method
- `PestManagement[]` — pest/disease identification, treatment, and prevention
- `HarvestCalendar[]` — first harvest timing, peak season, yield estimate
- `RevenueProjection[]` — year-by-year revenue projection per preset

This data is used by the planting guides UI section and by the AI advisor's response formatter.

### 33.9 Crop Library and Recommender API routers
**`backend/routers/crop_library_router.py`** — mounted at `/api/crops-v2`:
- `GET /api/crops-v2/all` — all 22 crop labels
